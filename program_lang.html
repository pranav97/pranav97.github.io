Multithreaded Languages Comparison

This project compares C++, C, Go and Rust for multithreading (backend implementation as well as syntax) using a simple matrix multiplication program written in all of these language.

Results:

The four programs ran on Hummingbird, which a compute cluster dedicated to UC Santa Cruz. The summary of the system configurations are: it has 32 CPUs and two threads per core, so a total of 64 logical cores and parallel lanes of execution for this algorithm. Also, the system has a much more powerful cache; 32kB of L1 data and instruction cache, 1024kB of L2 cache and 11264K of cache. Since matrix multiplication is demanding of the cache, the evaluation of this algorithm in machine with lessor cache would be meaningless. We were able to isolate the latency to just multithreading and not some of the other factors like cache.

Even after performing this experiement it is still difficult to say what the best language is because it depends on the use case. Go is excellent for smaller parallel scripts and larger ones as long as there is a testing framework that runs tools like "go test-race" to ensure that there are no multiple accesses or race conditions. Since Go prioritizes performance, Rust will not perform as well but will have a lot more guarantees. In highly available systems, Rust is the only option for any amount of complicated logic. Traditionally, reboots and reboots of components are used to handle a deadlock in the a unreponsive conponent. In highly available systems, a reboot of the system or the component is not desirable so the memory and thread safety that Rust offers really helps keep the deadlocks away and keeps the component chugging along. Even in terms of extensibility, it is a much smoother experience to add a new feature when there is a certain guarentee about not being able to introduce a race condition or a deadlock. C is the only option for a lot of embedded systems, and the performance is excellent. C++ and Rust are great options if the availability of tools is essential. C++ is a lot more unsafe but performs better.

In terms of multithreaded performance, the figure below shows that Go is the fastest followed by C, then C++, and finally, Rust. This graph has latency as the y axis which means the higher the y value is the worse the program performed. It is impossible to tell C, C++, and Go at a lower number of rows and columns. Rust is not the fastest, but it is the solution for highly available systems.

In terms of single-threaded performance, the fastest solution is still Go, followed by Rust, which tells us that these languages are close to the hardware as well. C is faster than C++ for single-threaded programs so.
